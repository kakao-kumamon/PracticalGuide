0. 시작하기 전에 — 이 가이드가 답하려는 것

요즘 Claude나 Codex 같은 AI에게 "버튼 하나 누르면 뭔가 만들어주는 도구"를 시켜본 사람이라면 한 번쯤 이런 경험을 해봤을 거다.

  • "플러그인 만들어줘" 한 줄에 코드는 잘 나오는데, 막상 돌려보면 안 된다
  • 고치다 보면 처음 만든 구조가 깨져서 점점 누더기가 된다
  • 그래픽 제너레이터를 만들었는데 결과물이 "AI가 만든 티"가 너무 난다

이 가이드는 그 간극을 좁히려는 문서다. "한 번에 잘 시키는 법"이 아니라, "여러 번 돌리면서 점점 좋아지게 만드는 워크플로우"가 핵심이다.

대상 독자는 코드를 약간 읽을 줄 알지만 직접 다 짜기는 부담스러운 사람—기획자, 디자이너, 개발 입문자, 또는 본업 외에 도구 하나 빠르게 만들고 싶은 누구나다.

1. 멘탈 모델: AI는 "주니어 동료"다

가장 큰 오해는 AI를 마법 상자로 보는 것이다. 사실 AI는 "빠르고 의욕 넘치지만 우리 프로젝트 맥락은 모르는 주니어 동료"에 가깝다. 이 멘탈 모델을 받아들이면 프롬프트 쓰는 방식이 자연스럽게 바뀐다.

마법 상자 관점주니어 동료 관점
한 번에 완벽하게 시킨다작업을 쪼개서 단계별로 시킨다
"플러그인 만들어줘""먼저 폴더 구조부터 잡자"
결과물을 그대로 쓴다리뷰하고, 의도 다른 부분 지적하고, 다시 시킨다
실패하면 AI가 멍청하다고 생각맥락을 덜 줬다고 생각하고 컨텍스트 보강
핵심

AI에게 "맥락을 얼마나 잘 떠먹여 주느냐"가 결과물 품질의 80%를 결정한다. 나머지 20%가 모델·프롬프트 테크닉이다.

2. 4단계 워크플로우 — 기획→설계→구현→검증

한 번에 코드부터 시키지 말고, 다음 네 단계를 분리해서 진행하면 결과물이 눈에 띄게 안정적이 된다.

① 기획 (Spec)

"무엇을 만들지"를 먼저 글로 정리한다. AI에게 시키는 게 아니라 내가 직접 한 페이지 내외로 적는 단계다. 들어가야 할 항목:

  • 한 줄 목적 — "이 도구는 ○○하는 사람을 위한 △△ 도구다"
  • 입력과 출력 — 사용자가 뭘 넣고, 결과로 뭐가 나오나
  • 제약 — 어디서 동작해야 하나(브라우저? 노션 플러그인? CLI?), 외부 API 쓰나
  • 안 할 것 (Non-goals) — 의외로 이게 가장 중요하다. AI가 과욕 부리는 걸 막아준다

② 설계 (Design)

이제 AI를 부른다. 단, 코드를 시키지 말고 설계 대화부터 한다.

위의 스펙을 보고 다음을 답해줘:
1) 이걸 구현할 때 가장 적합한 기술 스택과 그 이유
2) 폴더/파일 구조 제안
3) 가장 위험한 가정이나 빠진 요구사항 3가지
4) MVP로 쪼갠다면 어떤 순서로 만들어야 하나

여기서 AI가 짚어주는 "위험한 가정"이 진짜 보석이다. 종종 내가 놓친 케이스를 발견해준다.

③ 구현 (Build)

설계가 합의되면, 이제 작은 단위로 나눠서 시킨다. 한 번에 한 파일, 한 번에 한 기능이 원칙이다.

  • "먼저 색상 팔레트 추출 함수만 만들어줘. 입력은 이미지 파일 경로, 출력은 #hex 5개의 배열"
  • "그 함수를 쓰는 CLI 진입점을 만들어줘. argparse 쓰고 --output 옵션 추가"

이렇게 쪼개야 (a) 어디서 깨졌는지 디버깅이 쉽고, (b) AI가 컨텍스트를 잃지 않는다.

④ 검증 (Verify)

AI가 "다 됐어요!"라고 해도 절대 그대로 넘기지 말 것. 최소 두 가지를 직접 한다.

  1. 실제로 돌려본다. 데모 영상 만들 때처럼 처음부터 끝까지 시나리오를 따라간다.
  2. 일부러 이상한 입력을 넣어본다. 빈 값, 너무 큰 값, 한글, 이모지, 공백 — AI가 짠 코드는 의외로 엣지 케이스에 약하다.
자주 하는 실수

"보기엔 맞아 보여서" 그대로 commit하는 것. 코드 리뷰처럼 한 줄씩 읽으면서 "왜 이걸 이렇게 했지?"를 물어봐야 한다. 모르겠으면 AI에게 그 줄을 지목해서 설명을 시켜라.

3. 프롬프트 작성 패턴 7가지

실전에서 반복적으로 효과를 봤던 패턴들이다. 외우기보단 "필요할 때 꺼내 쓸 수 있게" 익혀두자.

패턴 1 — 역할·맥락·목표·출력형식 (RGCO)

4가지 구획으로 정리하면 빈약한 프롬프트가 한 단계 좋아진다.

[역할] 너는 Vanilla JS로 작은 브라우저 도구를 만드는 시니어 프론트 개발자야.
[맥락] 사내 디자이너들이 매주 SNS 카드 이미지 30장을 손으로 만들고 있어.
        템플릿은 4가지뿐이고 텍스트만 바뀌어.
[목표] CSV를 업로드하면 한 행당 PNG 한 장씩 자동 생성하는 페이지를 만들고 싶어.
[출력형식] 단일 HTML 파일. 외부 의존성 없이 작동해야 함. 한국어 폰트는 시스템 폰트 fallback.

패턴 2 — 거꾸로 시키기 (Show me what you'd ask)

스펙이 부족할 때 효과적이다.

이걸 구현하기 전에, 네가 나에게 물어봐야 할 결정적인 질문 5개를 먼저 알려줘. 코드는 아직 쓰지 마.

AI가 던지는 질문들이 곧 내 스펙의 빈틈이다.

패턴 3 — 비교 강제하기

AI는 첫 답에 만족시키도록 학습되어 있어서, 가만히 두면 그럴듯한 한 가지만 던진다. 강제로 대안을 만들게 하라.

접근법 3가지를 제시해줘. 각각 장단점과 언제 쓰는 게 좋은지. 그 다음 네 추천을 골라줘.

패턴 4 — 의심 표현 받기

구현 후에 추가로 시킨다.

방금 짠 코드에서 (a) 가정이 깨질 수 있는 부분, (b) 사용자가 예상 못 할 동작, (c) 6개월 뒤 내가 가장 후회할 결정 — 이렇게 세 영역을 비판적으로 짚어줘.

이 한 마디가 코드 리뷰 한 사이클을 대신해준다.

패턴 5 — 예시로 가르치기 (Few-shot)

스타일·톤·포맷처럼 말로 설명하기 어려운 것은 예시 2~3개를 직접 보여주는 게 빠르다.

이런 톤으로 카피를 만들어줘:
예시1: "오늘 안 사면 내일 더 비싸요" → 불안 + 짧은 문장
예시2: "이거 하나면 끝" → 단순함 강조
이제 카메라 광고 카피 5개 만들어줘.

패턴 6 — 점진적 정제

한 방에 만족스러운 결과는 거의 없다. 다음 형식으로 이어간다.

  • "좋은데, X만 바꿔줘. 나머지는 유지."
  • "방금 버전이 더 나아. 이걸 기준으로 Y만 추가."
  • "두 버전을 합쳐줘. 첫 번째의 구조 + 두 번째의 색감."

"나머지는 유지"라는 한 마디가 의외로 큰 힘을 발휘한다.

패턴 7 — 출력 컨테이너 지정

코드/문서를 받을 때는 어디에 넣을지를 미리 지정한다.

전체 파일을 index.html 한 덩어리로 줘. 부분 수정이면 변경된 함수만 따로 ```js 블록으로.

4. 실전 예시 ① 작은 플러그인 만들기

예시 시나리오: "브라우저에서 보고 있는 페이지의 한국어 본문을 영어로 자동 번역해서 옆에 붙여주는 크롬 확장"을 만든다고 하자.

Step 1. 스펙 한 페이지

목적
한국어 뉴스를 빠르게 영어로 훑고 싶은 사람을 위한 인라인 번역 익스텐션
입력
현재 탭의 본문 텍스트 (article 태그 우선)
출력
원문 옆에 영어 번역 사이드패널
제약
Manifest V3, 번역 API는 사용자 개인 키, 페이지 1회당 호출 1번
안 할 것
로그인, 결제, 텍스트 외 콘텐츠(이미지/비디오) 처리

Step 2. AI에게 설계 대화

위 스펙으로 Manifest V3 크롬 확장을 만들 거야.
1) 폴더 구조와 각 파일의 역할
2) content script vs background service worker 어디서 뭘 할지
3) API 키를 안전하게 저장하는 방법
4) MVP로 쪼개면 어떤 순서로 만들지
코드는 아직 쓰지 마. 위 4가지에 답만.

여기서 보통 AI가 chrome.storage 사용, manifest.json 권한 최소화 등 베이스라인 베스트프랙티스를 정리해준다. 이 답변을 그대로 README의 초안으로 저장해두면 나중에 다시 안 헤맨다.

Step 3. 한 파일씩 구현

  1. manifest.json만 만들기 → 직접 chrome://extensions에 로드해보기
  2. 옵션 페이지(options.html)에서 API 키 저장만 동작 확인
  3. content script로 본문 추출만 → console.log로 텍스트 잘 뽑히는지 확인
  4. background에서 API 호출 + content로 결과 송출 → 그제야 사이드패널 UI 작성
핵심 포인트

각 단계에서 "여기까지 작동해야 하는데 안 된다"가 명확해진다. AI에게 디버깅을 시킬 때 "사이드패널이 안 떠"보다 "background → content message가 도착은 하는데 DOM 삽입 시점에 article이 아직 없는 듯"이라고 정확히 진단해주면 해결률이 크게 오른다.

Step 4. 검증

실제로 자주 보는 사이트 5곳에서 돌려본다. 본문 추출 실패하는 사이트가 1~2곳은 반드시 나온다. 거기서 멈추지 말고 구조가 다른 페이지를 의도적으로 골라 테스트하는 게 핵심이다(예: 네이버 블로그, 미디엄, 노션 공개 페이지).

5. 실전 예시 ② 그래픽 제너레이터 만들기

예시 시나리오: "제목 + 짧은 설명을 입력하면 SNS용 1080x1080 카드 이미지를 PNG로 만들어주는 웹 도구".

왜 그래픽 도구가 까다로운가

그래픽 제너레이터는 코드보단 "보기에 좋다"가 결정한다. 그래서 일반 도구보다 한 단계 더 챙겨야 할 게 있다.

  • AI가 만든 첫 디자인은 거의 항상 "AI가 만든 티"가 난다 → 폰트, 여백, 색이 평범하다
  • 모듈화 없이 시키면 색·폰트·레이아웃이 한 덩어리로 묶여서 한 곳만 바꾸려 해도 다 깨진다

좋은 결과를 위한 5가지 처방

① 디자인 토큰을 먼저 정의

코드 짜기 전에 "팔레트, 타이포 스케일, 그리드"부터 확정한다. 직접 결정 못 하겠으면 AI에게 "Behance에서 인기 있는 카드 디자인 5가지의 공통점 분석"을 시킨 다음, 그 위에 결정한다.

② 캔버스는 한 함수에서만 그리게

HTML/Canvas로 짤 때, "render(state) 함수 하나만 캔버스를 건드리고, 나머지는 state만 바꾼다" 규칙을 프롬프트에 명시하라. 안 하면 AI가 곳곳에서 ctx를 만지는 코드를 쓴다.

③ 폰트·여백을 변수화

const T = {
  font: { title: 'Pretendard', body: 'Pretendard' },
  size: { title: 80, body: 32 },
  pad: { x: 64, y: 80 }
};

AI에게 "모든 수치는 T 객체를 통해서만 쓰고, 매직 넘버 금지"라고 못 박는다. 나중에 "전체적으로 좀 더 시원하게"라고만 시켜도 T값만 바뀌어서 깔끔하다.

④ 변형(Variants)을 분리

"기본 / 강조 / 인용" 같은 카드 종류는 함수가 아니라 설정으로 관리하라.

const variants = {
  default: { bg: '#fafaf7', accent: '#c97a3a', align: 'left' },
  emphasis: { bg: '#1a1a1a', accent: '#fdf6ec', align: 'center' },
  quote:    { bg: '#fdf6ec', accent: '#1a1a1a', align: 'left' }
};

새 변형 추가가 "객체 한 줄 추가"로 끝난다.

⑤ 미리보기와 결과물을 분리

실시간 미리보기는 빠르게(작은 캔버스), PNG 추출은 정확하게(원본 해상도). 한 캔버스로 두 마리 토끼 잡으려다 흐려진다.

점진적 개선 프롬프트 예시

좋아, 지금 카드 잘 나오는데 다음 두 가지만 고쳐줘:
1) 제목이 길 때 줄바꿈이 단어 중간에서 끊김 → 단어 단위 줄바꿈으로
2) 배경이 단색이라 단조로움 → variants에 'gradient' 추가, 다른 코드는 건드리지 마.
변경된 함수만 따로 보여줘.

6. AI가 헛다리 짚을 때 바로잡는 법

증상별 처방전

증상원인처방
코드는 그럴듯한데 실행하면 ImportError·UndefinedError 존재하지 않는 라이브러리·API를 환각 "이 코드에서 실제로 import하거나 호출한 모든 외부 의존성을 표로 만들고, 각각 공식 문서 URL을 명시해줘. 모르면 모른다고 답해."
고치라고 시킬수록 더 망가진다 이전 컨텍스트가 너무 길어 스코프 잃음 새 대화 시작. 정리된 전체 코드 + 한 줄짜리 요청만 다시 줌.
"다 됐어요"라는데 일부 기능 누락 요구사항이 길어 일부 흘림 요구사항을 번호 매겨 다시 주고, "각 항목별로 구현했는지 ✓/✗ 표시 후 ✗는 이유까지"
같은 버그를 계속 반복 AI가 같은 가정으로 같은 길을 감 "이 버그를 일으키는 가능한 원인 5가지를 가능성 순으로 나열. 그 다음 가장 가능성 낮은 것부터 검증법 알려줘."
UI가 "AI 티" 난다 레퍼런스 부재 좋아하는 사이트 스크린샷 2장 첨부 + "이런 톤으로 다시"

7. 체크리스트 — 출시 전 마지막 점검

기능 점검

  • 스펙의 입출력이 모두 동작하는가
  • "안 할 것 (Non-goals)"이 정말 안 들어갔는가 (AI가 자주 슬쩍 추가함)
  • 빈 값 / 비정상 값 / 한글·이모지 / 매우 큰 입력 4종 테스트했는가
  • 실패 시 사용자에게 보이는 메시지가 사람 말 같은가 ("Error 500" 금지)

코드 위생

  • 매직 넘버·하드코딩된 문자열이 상수로 빠져 있는가
  • API 키 같은 비밀이 코드에 박혀 있지 않은가
  • 외부 라이브러리는 실제로 존재하고 활발히 유지보수되는가
  • 주석이 코드를 그냥 따라 쓰지 않고 "왜"를 설명하는가

그래픽 도구 한정

  • 한국어·영어·이모지·긴 제목 4가지 케이스가 모두 깨지지 않는가
  • 다운로드된 PNG가 미리보기와 일치하는가 (해상도·여백)
  • 최소 3가지 변형이 각각 보기 좋은가

사용자 인계

  • 처음 쓰는 사람이 README만 보고 5분 안에 동작시킬 수 있는가
  • 예제 입력·예제 출력이 1세트 이상 포함되어 있는가
  • "이 도구가 안 하는 것"이 README 상단에 적혀 있는가

마지막 한마디. AI로 도구 만들기에서 가장 큰 함정은 한 번에 끝내려는 욕심이다. 한 사이클을 돌리고, 결과를 보고, 다음 한 가지만 고치는 — 이 리듬이 결국 가장 빠르다. 본 가이드의 워크플로우와 패턴들도 처음엔 번거롭게 느껴질 수 있지만, 두세 번 적용해보면 "AI 없이는 못 돌아가지만 AI에 끌려다니지도 않는" 자기만의 작업 루틴이 생긴다.
